# 产品需求文档 (Product Specification)

## 项目概述

**项目名称**：猜人名PK（guess-name-pk3）

**一句话描述**：一款双人对战猜人名 Web 游戏，玩家通过向 AI 提问是/否问题来缩小范围，最终竞速猜出隐藏的人物名字。

**目标**：打造一款轻量级、无需注册、即开即玩的双人本地对战 Web 猜名游戏，利用 AI 大模型实现智能出题和判定，提供沉浸式的竞技体验和精美的视觉效果。

---

## 目标用户

**核心用户群体**：喜欢聚会/休闲小游戏的年轻人及家庭用户

**用户特征**：
- 年龄段：12-45 岁
- 技术水平：零技术门槛，会使用浏览器即可
- 使用场景：朋友聚会、家庭娱乐、同事休闲、情侣互动
- 痛点：缺少一款免安装、多人面对面较量知识面的趣味小游戏

## 核心功能

### 功能 1：游戏主界面（Game Main View）

**功能描述**：游戏的核心界面，展示双方玩家信息、PK 对战区域和操作入口，是用户的主要交互界面。

**界面布局**：

```
┌──────────────────────────────────────────────────────┐
│ [游戏设置] [重新开始] [全屏显示]          猜人名PK    │
├──────────────────────────────────────────────────────┤
│                  限定范围: 不限                       │
│                                                      │
│    🐱 甜大官          VS          🐶 万小布           │
│   已提问: 3/12                   已提问: 5/12        │
│                                                      │
├─────────────────────┬────────────────────────────────┤
│   甜大官 的提问区     │     万小布 的提问区             │
│                     │                                │
│  Q1: 是男性吗？      │  Q1: 是中国人吗？               │
│  A1: ✅ 正确         │  A1: ❌ 错误                   │
│                     │                                │
│  Q2: 是演员吗？      │  Q2: 是歌手吗？                │
│  A2: ❌ 错误         │  A2: ✅ 正确                   │
│                     │                                │
│  ┌───────────────┐  │  ┌───────────────┐            │
│  │  输入提问...    │  │  │  输入提问...    │            │
│  └───────────────┘  │  └───────────────┘            │
│  [是否提问] [猜测结果] │  [是否提问] [猜测结果]          │
│  [放弃认输]          │  [放弃认输]                     │
└─────────────────────┴────────────────────────────────┘
```

**输入**：
- 页面加载时自动读取本地存储的游戏设置

**输出**：
- 完整渲染的游戏主界面

**业务规则**：
- 页面左上角固定显示三个按钮：「游戏设置」「重新开始」「全屏显示」
- 右上角或顶部中央显示游戏标题"猜人名PK"
- 中部显示两位玩家的名字、emoji 头像、当前提问次数/最大提问次数
- 如果设置了人名限定范围（如"中国古代诗人"），在显著位置展示该限定范围
- 下方分左右两栏，分别展示两位玩家的提问记录和操作区
- 每个玩家的操作区包括：输入框（最多 20 个汉字）、「是否提问」按钮、「猜测结果」按钮、「放弃认输」按钮
- 两位玩家的操作**相互独立**，不是回合制轮流制，各自可随时操作
- 首次进入页面时，需要先点击「游戏设置」完成 API Key 配置，然后才能开始游戏

**异常处理**：
- 未设置 API Key 时，点击提问或猜测按钮弹出提示引导配置
- 页面加载失败时显示重试提示

**优先级**：高

**状态**：待开发

### 功能 2：游戏设置（Game Settings）

**功能描述**：通过弹窗/模态框形式提供游戏的各项参数设置，包括玩家信息、出题范围、提问次数限制和 AI 大模型配置。

**输入**：
- 玩家姓名：字符串，最多 4 个字符
- Emoji 头像：从 50 个预设图标中选择
- 人名出题范围：下拉菜单选择
- 最大提问次数：数值，默认 12
- AI 大模型 API Key：字符串
- AI 大模型提供商：下拉菜单选择

**输出**：
- 设置保存到 localStorage
- 界面实时更新对应设置

**业务规则**：

#### a) 玩家信息设置
- 支持设置两位玩家的名字和 emoji 头像
- 名字最多 4 个字符（中文、英文均可）
- 默认名字：玩家1 = "甜大官"，玩家2 = "万小布"
- Emoji 头像提供 50 个预设图标供选择
- 图标要求：带有不同色彩风格的圆形底纹背景，包含多种类别（动物、球类、工具、表情、食物、植物等）
- 默认头像：玩家1 = 🐱（猫），玩家2 = 🐶（狗）
- 两位玩家不可选择相同头像，选择重复时弹出提示

#### b) 人名出题范围设置
- 通过下拉菜单选择，选项列表如下（排列顺序即菜单顺序）：
  1. 不限（默认）
  2. 限中国知名人物
  3. 限全球知名人物
  4. 限全球虚拟人物
  5. 限中国演员歌手
  6. 限中国古代诗人
  7. 限三国演义人物
  8. 限中国知名皇帝
  9. 限马云马化腾张一鸣之一
- 所有选项的核心要求：生成的名字必须是绝大多数人所熟知的，不得是冷僻偏门小众人物

#### c) 最大提问次数设置
- 输入框或滑块控件
- 有效范围：5 ~ 30 次
- 默认值：12 次
- 包含"是否提问"和"猜测结果（猜错时）"在内的所有提问

#### d) AI 大模型设置
- 支持设置 API Key
- 支持选择大模型提供商，初始支持：
  - **智谱 AI（GLM）**（默认）
  - **Google Gemini**（可扩展）
- API Key 输入框使用密码遮罩显示，带"显示/隐藏"切换按钮
- API Key 保存在 localStorage 中

#### e) 恢复默认设置
- 提供「恢复默认设置」按钮
- 恢复后：玩家名字归位、头像归位、出题范围恢复"不限"、最大提问次数恢复 12、API Key 保留不清空
- 恢复前需弹窗二次确认

**异常处理**：
- 玩家名字为空时不允许保存，提示"请输入玩家名字"
- 玩家名字超长自动截断
- API Key 格式校验（非空，长度合理）
- 设置保存失败（localStorage 满或不可用）时提示用户

**优先级**：高

**状态**：待开发

### 功能 3：AI 智能出题（AI Name Generation）

**功能描述**：游戏开始或重新开始时，AI 根据设定的出题范围分别为两位玩家生成要猜测的人物名字。

**输入**：
- 出题范围设置（如"限中国古代诗人"）
- 两位玩家的标识（确保生成不同的名字）

**输出**：
- 为玩家1生成的人物名字（字符串）
- 为玩家2生成的人物名字（字符串）

**业务规则**：
- 两位玩家的人物名字必须**不相同**
- 生成的人物名字必须是**绝大多数人都熟知**的，不得冷僻偏门
- 人物名字存储在内存中，不展示给玩家
- 只有在玩家**选择放弃认输**后，才向该玩家揭晓其对应的人物名字
- 猜测正确后，也应揭晓该人物名字（确认展示）
- AI 出题的 Prompt 设计需包含以下约束：
  - "请根据以下范围生成一个知名人物名字：{范围}"
  - "该人物应该是绝大多数中文用户都知道的"
  - "请只返回人物名字，不要其他内容"
  - "注意：本次生成的名字不可以是 {已有名字}"
- 对于"限马云马化腾张一鸣之一"选项，直接从 ["马云", "马化腾", "张一鸣"] 中随机选取，无需调用 AI

**异常处理**：
- AI 返回空值或无效内容：重试最多 3 次，仍失败则提示用户
- AI 返回冷僻人物时无法自动校验，依赖 Prompt 约束
- 网络请求超时（10秒）：显示超时提示，允许重试
- API 配额耗尽：显示明确错误信息，引导用户更换 Key 或等待

**优先级**：高

**状态**：待开发

### 功能 4：AI 提问判定（AI Question Judging）

**功能描述**：玩家提出"是否问题"或"猜测人名"时，AI 根据隐藏的答案进行判定，只返回"正确"或"错误"。

**输入**：
- 玩家输入的问题/猜测文本（字符串，不超过 20 个汉字）
- 当前隐藏的人物名字（AI 判定的依据）
- 提问类型：「是否提问」或「猜测结果」

**输出**：
- 判定结果：「✅ 正确」或「❌ 错误」

**业务规则**：

#### "是否提问"模式
- 玩家提问必须是可以用"是/否"回答的问题
- AI 判定 Prompt 设计：
  - "现在有一个人物名叫 {人名}。用户问：{问题}。请根据事实判断这个问题是正确还是错误。只回答'正确'或'错误'两个字。"
- AI 仅允许返回"正确"或"错误"
- 如果 AI 返回了其他内容，前端解析逻辑：包含"正确"→ 正确，包含"错误"→ 错误，都不包含 → 重试

#### "猜测结果"模式
- 玩家输入一个人物名字进行猜测
- AI 判定 Prompt 设计：
  - "答案人物名字是 {人名}。用户猜测的是：{猜测内容}。请判断用户猜测是否正确。只回答'正确'或'错误'两个字。注意：即使用户使用了别名、外号或部分名字，只要能明确指向同一人物，就算正确。"
- 猜测正确 → 该玩家获胜，游戏标记该玩家完成
- 猜测错误 → 计为一次提问消耗，记录在提问历史中

**通用规则**：
- 每次提问（无论是否提问还是猜测错误）都计为一次提问，计入已用次数
- 达到最大提问次数后，仅允许使用「猜测结果」功能，不再允许「是否提问」
- 若达到最大提问次数且猜测仍然失败，自动视为该玩家失败（等同于放弃）
- 提问记录按时间顺序展示，最新的在最下方
- 每条记录格式：`Q{序号}: {问题内容}` + `A{序号}: ✅ 正确 / ❌ 错误`

**异常处理**：
- 用户输入为空：按钮禁用或提示"请输入问题"
- 用户输入超过 20 个汉字：自动截断并提示
- AI 接口调用失败：显示"判定失败，请重试"，不消耗提问次数
- AI 返回异常内容（既非"正确"也非"错误"）：重试 2 次，仍异常则提示用户
- 网络超时：显示超时提示，允许重试，不消耗提问次数

**优先级**：高

**状态**：待开发

### 功能 5：胜负判定与游戏结束（Win/Lose & Game Over）

**功能描述**：管理游戏的胜负逻辑和游戏结束状态。

**输入**：
- 两位玩家的当前状态（进行中/已猜对/已放弃/已达上限）

**输出**：
- 胜负判定结果
- 游戏结束界面

**业务规则**：

#### 单个玩家的状态流转
```
进行中 → 猜测正确（胜利）
进行中 → 主动放弃（失败）
进行中 → 提问次数达上限 + 最后猜测失败（失败）
```

#### 整场游戏结束条件
- 两位玩家**都不再是"进行中"**时，整场游戏才结束
- 即：一位玩家猜对或放弃后，另一位可继续游戏

#### 胜负结果展示
| 玩家1状态 | 玩家2状态 | 结果展示 |
|----------|----------|---------|
| 猜对 | 猜对 | 双方都猜对！用时少/提问少的玩家显示为优胜 |
| 猜对 | 放弃/失败 | 玩家1 🏆 胜利！ |
| 放弃/失败 | 猜对 | 玩家2 🏆 胜利！ |
| 放弃/失败 | 放弃/失败 | 双方都未猜出，平局 |

#### 游戏结束界面
- 弹出结果弹窗，显示：
  - 胜负结果（带动画效果，如🎉撒花）
  - 双方各自要猜的人物名字（揭晓）
  - 双方的提问次数统计
  - 「重新开始」和「游戏设置」按钮

#### 放弃认输机制
- 每位玩家的操作区下方有「放弃认输」按钮
- 点击后弹出二次确认："确定要放弃吗？放弃后将揭晓你的答案。"
- 确认放弃后：
  - 揭晓该玩家对应的人物名字
  - 该玩家的操作区锁定（所有输入框和按钮禁用）
  - 另一位玩家可继续操作

**异常处理**：
- 页面意外刷新：当前游戏状态丢失，提示用户重新开始（v1 不做状态持久化）

**优先级**：高

**状态**：待开发

### 功能 6：重新开始（Restart Game）

**功能描述**：在不改变游戏设置的情况下，清空双方提问记录并重新出题。

**输入**：
- 用户点击「重新开始」按钮

**输出**：
- 双方提问记录清空
- 重新调用 AI 生成两个新的人物名字
- 游戏状态重置为"进行中"

**业务规则**：
- 保留的设置：玩家名字、emoji 头像、出题范围、最大提问次数、API Key、大模型提供商
- 清空的内容：双方所有提问记录和判定结果、胜负状态、已用提问次数
- 重新生成：两个新的人物名字
- 游戏进行中点击「重新开始」需弹出二次确认："当前游戏尚未结束，确定要重新开始吗？"
- 游戏已结束时点击「重新开始」直接执行，无需确认

**异常处理**：
- 重新出题（AI调用）失败：保持当前状态不变，提示用户重试

**优先级**：高

**状态**：待开发

### 功能 7：全屏显示（Fullscreen Toggle）

**功能描述**：切换浏览器全屏模式，提供更沉浸的游戏体验。

**输入**：
- 用户点击「全屏显示」按钮

**输出**：
- 切换全屏/退出全屏

**业务规则**：
- 使用浏览器 Fullscreen API（`document.documentElement.requestFullscreen()`）
- 全屏时按钮文字切换为"退出全屏"
- 支持 ESC 键退出全屏
- 全屏状态下界面自适应全屏分辨率

**异常处理**：
- 浏览器不支持全屏 API：隐藏该按钮或点击后提示不支持

**优先级**：低

**状态**：待开发

### 功能 8：音效系统（Sound Effects）

**功能描述**：为游戏中的各种交互操作提供音效反馈，增强游戏沉浸感。

**业务规则**：

| 交互动作 | 音效类型 | 说明 |
|---------|---------|------|
| 按钮点击 | 短促清脆的点击音 | 所有按钮通用 |
| 提交提问 | 发送/弹出声 | 提问发送时播放 |
| 判定"✅ 正确" | 正面清脆音效（叮） | 得到正确答案反馈 |
| 判定"❌ 错误" | 负面低沉音效（嗡） | 得到错误答案反馈 |
| 猜测成功（胜利） | 胜利号角/撒花音效 | 猜对人名时播放 |
| 放弃认输 | 失败/遗憾音效 | 玩家放弃时播放 |
| 游戏开始/重新开始 | 开场鼓点/启动音 | 开始新游戏时播放 |
| 打开设置弹窗 | 清脆开启音 | 设置面板打开时 |

- 音效使用 Web Audio API 或 `<audio>` 元素实现
- 音效文件采用内嵌 Base64 数据或合成音的方式，避免外部依赖
- **可选**：提供静音/取消静音的小图标按钮（放在页面右上角）

**优先级**：中

**状态**：待开发

### 功能 9：动态视觉效果（Dynamic Visual Effects）

**功能描述**：为游戏界面添加丰富的动画和视觉效果，使游戏体验更加生动有趣。

**业务规则**：

| 效果类型 | 说明 |
|---------|------|
| 背景动画 | 深色渐变背景 + 缓慢流动的粒子/光斑效果 |
| VS 区域动画 | "VS" 文字带有脉冲发光效果 |
| 按钮悬停效果 | 悬停时发光/放大，带有过渡动画 |
| 提问提交动画 | 新提问出现时的滑入/淡入效果 |
| 判定结果动画 | ✅/❌ 出现时的弹簧/抖动效果 |
| 猜对庆祝动画 | 全屏粒子撒花 + 🎉 emoji 飞散效果 |
| 放弃时效果 | 该玩家区域变灰 + 轻微的失败动画 |
| 计数器动画 | 提问次数增加时的数字翻转/弹跳效果 |
| 头像状态动画 | 进行中时头像有呼吸灯效果，胜利时有光环 |
| 加载等待动画 | AI 判定中时显示"思考中..."的跳动圆点动画 |

- 所有动画使用 CSS Animation / CSS Transition 实现，避免使用重型动画库
- 动画不影响页面交互性能
- 遵守 `prefers-reduced-motion` 偏好设置

**优先级**：中

**状态**：待开发

## 功能优先级

### 高优先级（必须有）
1. 游戏主界面 - 核心交互界面，是整个游戏的基础
2. 游戏设置 - 配置功能是游戏正常运行的前提
3. AI 智能出题 - 没有出题游戏无法进行
4. AI 提问判定 - 核心游戏逻辑
5. 胜负判定与游戏结束 - 完整的游戏循环
6. 重新开始 - 游戏可重复性

### 中优先级（应该有）
1. 音效系统 - 大幅提升游戏体验
2. 动态视觉效果 - 使游戏具有高级感
3. 响应式布局 - 支持移动端访问

### 低优先级（可以有）
1. 全屏显示 - 锦上添花功能
2. 游戏状态持久化 - 刷新后恢复游戏状态（v2考虑）
3. 游戏历史战绩记录 - 统计历史胜负（v2考虑）

## AI增强功能

本产品的核心玩法完全依赖 AI 大模型，以下是 AI 增强功能的详细设计：

### AI增强 1：智能出题引擎

**AI能力描述**：根据限定范围自动生成知名度高的人物名字，保证游戏可玩性。

**触发条件**：游戏开始或重新开始时自动触发

**输入**：出题范围设置、已生成的另一位玩家的人名（去重用）

**输出**：一个人物名字字符串

**Prompt 模板**：
```
你是猜人名游戏的出题官。请根据以下范围生成一个人物名字。

出题范围：{category}
排除名字：{exclude_name}（不可与此相同）

要求：
1. 该人物必须是绝大多数中文用户都知道的知名人物
2. 不得是冷僻偏门小众人物
3. 仅返回人物名字，不要任何其他文字、标点或解释

请直接输出人物名字：
```

---

### AI增强 2：智能是否判定

**AI能力描述**：理解玩家的自然语言提问，结合隐藏答案精准判定是/否。

**触发条件**：玩家点击「是否提问」按钮时触发

**输入**：隐藏的人物名字、玩家的提问文本

**输出**：仅"正确"或"错误"

**Prompt 模板**：
```
你是猜人名游戏的判定官。玩家正在猜一个人物，你需要根据事实判定玩家的提问。

隐藏的答案人物：{secret_name}
玩家提问：{question}

规则：
1. 根据关于"{secret_name}"的公认事实来判定
2. 如果提问内容关于该人物是正确的，回答"正确"
3. 如果提问内容关于该人物是错误的，回答"错误"
4. 你只能回答"正确"或"错误"这两个字，不得包含任何其他文字

请判定：
```

---

### AI增强 3：智能猜测判定

**AI能力描述**：判断玩家猜测的名字是否与答案匹配，支持别名、外号等模糊匹配。

**触发条件**：玩家点击「猜测结果」按钮时触发

**输入**：隐藏的人物名字、玩家猜测的名字

**输出**：仅"正确"或"错误"

**Prompt 模板**：
```
你是猜人名游戏的判定官。请判断玩家猜测的人名是否正确。

正确答案：{secret_name}
玩家猜测：{guess}

规则：
1. 如果玩家猜测与正确答案指向同一个人物，回答"正确"
2. 即使使用了别名、外号、艺名或简称，只要明确指向同一人物就算"正确"
3. 如果不是同一人物，回答"错误"
4. 你只能回答"正确"或"错误"这两个字

请判定：
```

## 非功能性需求

### 性能要求
- 页面首屏加载时间 ≤ 2 秒
- UI 交互响应时间 ≤ 100ms
- AI 判定请求的用户感知等待时间 ≤ 5 秒（含动画过渡）
- 动画帧率保持 60fps，避免卡顿

### 安全要求
- API Key 使用 localStorage 存储，不通过网络传输给第三方
- API Key 在页面上默认以密码遮罩形式展示
- 人物答案不存储在 DOM 中或全局变量中可读的地方（基本防窥屏）
- 所有 AI 请求使用 HTTPS

### 可用性要求
- 零注册、零安装，打开浏览器即可使用
- 界面操作直观，无需教程（但可考虑首次进入时简短引导）
- 所有按钮和交互元素有明确的视觉反馈
- 错误信息以友好的中文展示
- 支持静音模式

### 兼容性要求
- 现代浏览器支持：Chrome 90+、Firefox 90+、Safari 15+、Edge 90+
- 响应式布局：支持桌面端（1024px+）和平板端（768px+）
- 移动端（< 768px）：左右两栏改为上下两栏布局，或 Tab 切换模式
- 不依赖后端服务器（纯前端 + 直连 AI API）

## 技术栈建议

### 前端
- **HTML5 + CSS3 + JavaScript（原生）**：轻量级实现，不引入框架
- **CSS Variables**：全局设计系统变量（色彩、间距、圆角等）
- **CSS Grid / Flexbox**：响应式布局
- **CSS Animations / Transitions**：动态视觉效果
- **Google Fonts（Inter/Noto Sans SC）**：现代中文字体

### 后端
- 无后端。纯前端 SPA，直接调用 AI 大模型 API

### 数据库
- 无数据库。使用 localStorage 存储设置和临时状态

### AI 服务
- **智谱 AI（ZhipuAI / GLM）**：默认大模型提供商
  - API 端点：`https://open.bigmodel.cn/api/paas/v4/chat/completions`
  - 模型：`glm-4-flash`（响应快、成本低）
- **Google Gemini**（可选扩展）
  - API 端点：`https://generativelanguage.googleapis.com/v1beta/models/gemini-pro:generateContent`

### 音效
- **Web Audio API**：通过代码合成简单音效（无需外部音频文件）
- 或使用极小的内嵌 Base64 音频数据

### 部署
- 纯静态文件，支持通过 GitHub Pages、Vercel、或直接本地双击 HTML 文件使用
